# 分布式锁学习笔记 - 第四阶段

## 一、为什么需要分布式锁

### 1.1 单机锁的局限

在单机应用中，我们可以用 Java 的 `synchronized` 或 `ReentrantLock` 来保证线程安全：

```java
// 单机锁（只在当前JVM有效）
synchronized (this) {
    // 临界区代码
}
```

**问题**：当应用部署到多台服务器时，每个 JVM 都有自己的一把锁，**单机锁无法跨 JVM 生效**。

### 1.2 分布式锁的需求

```
场景：用户A同时在两台服务器上发起秒杀请求

服务器1（JVM-1）：synchronized(this) → 获得锁 → 执行业务
服务器2（JVM-2）：synchronized(this) → 获得锁 → 执行业务  ← 两个锁互不影响！

结果：同一用户在同一时刻成功下单两次 → 超卖！
```

**分布式锁**：所有 JVM 共享同一把锁，保证同一时刻只有一个线程能执行临界区代码。

```
服务器1（JVM-1）：获取分布式锁 → 获得 → 执行业务 → 释放锁
服务器2（JVM-2）：获取分布式锁 → 等待... → 获得锁 → 执行业务

结果：同一用户同一时刻只有一个请求能执行 → 不会超卖
```

## 二、Redisson 简介

### 2.1 什么是 Redisson

Redisson 是一个在 Redis 基础上的 Java 高级协调工具包，提供了：

| 功能 | 说明 |
|------|------|
| **分布式锁** | 可重入锁、公平锁、读写锁、联锁 |
| **分布式集合** | Map、Set、Queue、Deque |
| **分布式服务** | 分布式任务调度、缓存、消息 |
| **看门狗机制** | 自动续期，防止锁过期 |

### 2.2 为什么不用原生 Redis

| 对比项 | 原生 Redis (setnx) | Redisson |
|--------|-------------------|----------|
| **实现复杂度** | 高（自己处理续期、异常） | 低（开箱即用） |
| **锁续期** | 需要自己实现 | 看门狗自动续期 |
| **可重入** | 不支持 | 支持 |
| **公平锁** | 不支持 | 支持 |
| **异常处理** | 需要自己处理 | 内置完善 |

## 三、核心概念

### 3.1 看门狗（Watchdog）

**问题**：如果持有锁的进程宕机了，锁永远不会释放，怎么办？

**Redisson 的解决方案**：看门狗

```
正常流程：
1. 线程获取锁，默认持有30秒
2. 业务执行中...（每10秒续期一次）
3. 业务完成，手动释放锁

宕机流程：
1. 线程获取锁，默认持有30秒
2. 业务执行中...（每10秒续期一次）
3. 进程宕机！锁不再续期
4. 30秒后锁自动过期，其他线程可以获取
```

### 3.2 锁的参数

```java
// 尝试获取锁
boolean tryLock(long waitTime, long leaseTime, TimeUnit unit);
//        等待时间    持有时间      时间单位
```

| 参数 | 说明 | 建议值 |
|------|------|--------|
| `waitTime` | 最多等待多久获取锁 | 5秒 |
| `leaseTime` | 持有锁的最长时间（-1=看门狗） | -1（生产推荐） |
| `unit` | 时间单位 | `TimeUnit.SECONDS` |

### 3.3 lock 方法详解（打饭排队比喻）

#### lock() — 无条件排队

```java
lock.lock();  // 什么都不传
```

```
场景：你去食堂打饭
  你：我要打饭
  食堂：排队吧，不知道排多久，反正排到你就给你打
  你：一直站着等...（可能等1分钟，可能等1小时）

特点：
  - 不会超时，一直等到拿到锁
  - 没有看门狗，锁不会自动续期
  - 如果持有锁的人宕机了，你永远等不到
```

#### lock(30, SECONDS) — 限时排队

```java
lock.lock(30, TimeUnit.SECONDS);  // 只等30秒
```

```
场景：你去食堂打饭
  你：我要打饭，最多等30秒
  食堂：排队吧
  你：等了30秒还没轮到 → 走了，不排了

特点：
  - 最多等30秒，拿不到锁就放弃
  - 没有看门狗，锁30秒后自动过期
  - 适合：业务执行时间确定，不想等太久
```

#### tryLock(5, 30, SECONDS) — 限时排队 + 锁有保质期

```java
lock.tryLock(5, 30, TimeUnit.SECONDS);
//  第1个参数：最多等5秒获取锁
//  第2个参数：拿到锁后最多持有30秒
```

```
场景：你去食堂打饭
  你：我要打饭，最多等5秒
  食堂：排队吧
  你：5秒内轮到了 → 打饭
  你：5秒内没轮到 → 走了

  你：打饭过程中，最多打30秒
  你：30秒内打完 → 走了
  你：30秒内没打完 → 食堂把饭收回，别人可以打了

特点：
  - waitTime=5：获取锁最多等5秒
  - leaseTime=30：拿到锁后最多持有30秒
  - 没有看门狗，30秒后锁自动过期
```

#### tryLock(5, -1, SECONDS) — 限时排队 + 看门狗护航（最推荐）

```java
lock.tryLock(5, -1, TimeUnit.SECONDS);
//  第1个参数：最多等5秒获取锁
//  第2个参数：-1表示启用看门狗
```

```
场景：你去食堂打饭
  你：我要打饭，最多等5秒
  食堂：排队吧
  你：5秒内轮到了 → 打饭

  你：打饭过程中...
  食堂：每10秒问你一句"还在吃吗？"
  你：在 → 食堂给你续时间（重置为30秒）
  你：不回答了（宕机了）→ 30秒后饭被收回

特点：
  - waitTime=5：获取锁最多等5秒
  - leaseTime=-1：启用看门狗，锁会自动续期
  - 只要线程还活着，锁永远不会过期
  - 线程宕机了，30秒后锁自动过期
```

#### 对比总结

| 方法 | 获取锁等待时间 | 锁持有时间 | 看门狗 | 适用场景 |
|------|---------------|-----------|--------|---------|
| `lock()` | 无限等待 | 无限（不释放就死锁） | 无 | 确定能拿到锁 |
| `lock(30, SECONDS)` | 无限等待 | 30秒 | 无 | 限时执行 |
| `tryLock(5, 30, SECONDS)` | 最多5秒 | 30秒 | 无 | 超时放弃 + 限时执行 |
| `tryLock(5, -1, SECONDS)` | 最多5秒 | 自动续期 | 有 | **生产推荐** |

### 3.4 可重入锁

同一个线程可以多次获取同一把锁：

```java
RLock lock = redissonClient.getLock("myLock");
lock.lock();
try {
    // 业务逻辑1
    lock.lock();  // 同一线程可以再次获取
    try {
        // 业务逻辑2
    } finally {
        lock.unlock();
    }
} finally {
    lock.unlock();
}
```

```
   ↓
┌─────────────────────────────────────────────────────┐

│    ├─ 发送成功 → 返回"排队中，订单号：xxx"             │

└─────────────────────────────────────────────────────┘
   ↓
┌─────────────────────────────────────────────────────┐
│ 5. 释放锁（finally块）                               │
│    if (lock.isHeldByCurrentThread()) {              │

│    }                                                 │
└─────────────────────────────────────────────────────┘
```

## 五、V1 vs V2 对比

### 5.1 架构对比

```
V1（Lua脚本方案）：
  请求 → Lua脚本（原子操作：检查+扣减+标记）→ 发MQ → 返回
  特点：无锁，靠Redis单线程保证原子性

V2（分布式锁方案）：
  请求 → 获取锁 → 检查+扣减 → 释放锁 → 发MQ → 返回
  特点：加锁，保证并发安全
```

### 5.2 代码对比

| 对比项 | V1（Lua脚本） | V2（分布式锁） |
|--------|---------------|----------------|
| **核心机制** | Redis Lua脚本原子操作 | Redisson RLock |
| **锁的使用** | 无锁 | `tryLock()` + `unlock()` |
| **并发控制** | 靠Redis单线程 | 靠分布式锁 |
| **代码复杂度** | 低（一个Lua脚本） | 中（锁+异常处理） |
| **性能** | 高（一次Redis调用） | 中（多次Redis调用） |
| **适用场景** | 简单秒杀 | 复杂业务逻辑 |

### 5.3 适用场景

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| **简单秒杀** | V1（Lua脚本） | 性能高，代码简单 |
| **复杂业务** | V2（分布式锁） | 业务逻辑复杂，Lua脚本写不下 |
| **多资源操作** | V2（分布式锁） | 需要同时操作多个Redis key |
| **高并发** | V1（Lua脚本） | 减少Redis调用次数 |

## 六、分布式锁的常见问题

### 6.1 锁超时问题

**问题**：如果业务执行时间超过锁的持有时间，锁会自动过期，其他线程可以获取锁。

**解决方案**：
1. **看门狗自动续期**（Redisson默认）
2. **设置合理的leaseTime**（比业务最大执行时间长）
3. **业务完成前检测锁是否还持有**

### 6.2 锁误删问题

**问题**：线程A的锁过期了，线程B获取了锁，然后线程A执行完删除锁，把线程B的锁删掉了。

**解决方案**：
1. **Redisson已解决**：锁的值是UUID，只删除自己加的锁
2. **手动实现**：删除锁时检查锁的值是否是自己设置的

### 6.3 可重入问题

**问题**：同一个线程需要多次获取同一把锁。

**解决方案**：
1. **Redisson已解决**：支持可重入锁
2. **手动实现**：用Hash结构存储锁信息，记录重入次数

## 八、Redis 序列化问题

### 8.1 为什么要注意序列化

Redis 存储的是字节数据，Java 对象需要序列化后才能存入 Redis。不同的序列化方式会影响：

- **兼容性**：不同版本的序列化格式可能不兼容
- **性能**：序列化/反序列化的速度
- **体积**：存储数据的大小
- **可读性**：能否直接查看 Redis 中的数据

### 8.2 三种序列化方式对比

| 对比项 | Kryo（二进制） | JSON（文本） | StringRedisTemplate |
|--------|---------------|--------------|---------------------|
| **格式** | 二进制字节 | JSON文本 | 纯字符串 |
| **可读性** | ❌ 乱码 | ✅ 人可读 | ✅ 人可读 |
| **跨版本** | ❌ 容易不兼容 | ✅ 兼容 | ✅ 兼容 |
| **跨语言** | ❌ 仅Java | ✅ 任意语言 | ✅ 任意语言 |
| **性能** | 最快 | 中等 | 中等 |
| **体积** | 最小 | 中等 | 中等 |

### 8.3 项目中的应用

**V1（StringRedisTemplate）**：
```java
// 存的是字符串，没有序列化问题
stringRedisTemplate.opsForValue().set(stockKey, "100");
stringRedisTemplate.opsForValue().get(stockKey);  // 返回字符串"100"
```

**V2（Redisson + JSON）**：
```java
// 使用JSON序列化，避免Kryo兼容性问题
config.setCodec(new JsonJacksonCodec());
// 存入对象会被序列化为JSON格式
redissonClient.getBucket(stockKey).set(100L);
```

### 8.4 常见错误及解决

**错误：KryoException: Encountered unregistered class ID**

```
原因：Redis 里有旧版本 Kryo 序列化的数据，新版本无法识别
解决：
  1. 清空 Redis：redis-cli FLUSHALL
  2. 改用 JSON 序列化：config.setCodec(new JsonJacksonCodec())
```

### 8.5 生产环境建议

```
简单场景（数字/字符串）：
  → StringRedisTemplate（V1用的）

复杂场景（Java对象）：
  → Redisson + JsonJacksonCodec（V2现在用的）

追求性能：
  → Redisson + KryoCodec（但要管理好版本兼容性）
```
