# 限流功能实现说明

## 一、概述

本项目使用 **Guava RateLimiter（令牌桶算法）+ 自定义注解 + AOP切面** 实现声明式限流，仅需在方法上加一行注解即可生效。

**技术栈**：Guava RateLimiter + Spring AOP

**核心文件**：

| 文件 | 职责 |
|------|------|
| `annotation/RateLimit.java` | 限流注解，声明QPS和提示信息 |
| `aspect/RateLimitAspect.java` | AOP切面，拦截所有@RateLimit方法 |
| `service/RateLimitService.java` | 限流服务，管理令牌桶和令牌获取 |

---

## 二、整体架构

```
请求进来
   ↓
SeckillController.seckill()  ← 方法上标注了 @RateLimit(qps = 100)
   ↓
RateLimitAspect.around()     ← AOP拦截所有@RateLimit注解的方法
   ↓
RateLimitService.tryAcquire() ← 真正判断是否限流（Guava RateLimiter）
   ↓
┌──────────────────────────────────────┐
│  有令牌？                             │
│    是 → 放行，执行业务方法            │
│    否 → 抛出 IllegalStateException    │
└──────────────────────────────────────┘
```

## 三、第一层：@RateLimit 注解

**文件**：`annotation/RateLimit.java`

```java
@Target(ElementType.METHOD)         // 只能标注在方法上
@Retention(RetentionPolicy.RUNTIME) // 运行时保留，AOP能读到
@Documented
public @interface RateLimit {
    
    String value() default "";                              // 接口标识，默认用"类名.方法名"
    double qps() default 100;                               // 每秒允许多少请求
    long timeoutMs() default 0;                             // 等多久拿令牌，0=不等直接拒绝
    String message() default "系统繁忙，请稍后重试";         // 被限流时的提示信息
}
```

### 参数说明

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `value` | String | `""` | 接口标识，为空时自动使用"类名.方法名" |
| `qps` | double | `100` | 每秒允许通过的请求数（Queries Per Second） |
| `timeoutMs` | long | `0` | 获取令牌的等待时间，0表示不等待直接拒绝 |
| `message` | String | `"系统繁忙，请稍后重试"` | 被限流时返回给用户的提示 |

### 使用方式

在 Controller 方法上加一个注解就完事了：

```java
@GetMapping("/{productId}")
@RateLimit(qps = 100, message = "请求过于频繁，请稍后重试")
public Map<String, Object> seckill(@PathVariable Long productId,
                                   @RequestParam(defaultValue = "1") Long userId) {
    // 业务逻辑...
}
```

## 四、第二层：RateLimitAspect 切面

**文件**：`aspect/RateLimitAspect.java`

AOP切面执行在 Controller 方法**之前**，如果被限流，业务代码根本不会执行。

### 核心代码

```java
@Aspect
@Component
public class RateLimitAspect {

    @Autowired
    private RateLimitService rateLimitService;

    @Around("@annotation(com.example.seckill.annotation.RateLimit)")
    public Object around(ProceedingJoinPoint joinPoint) throws Throwable {
        // 1. 获取注解信息
        MethodSignature signature = (MethodSignature) joinPoint.getSignature();
        Method method = signature.getMethod();
        RateLimit rateLimit = method.getAnnotation(RateLimit.class);
        
        // 2. 获取接口标识
        String key = rateLimit.value();
        if (key.isEmpty()) {
            key = method.getDeclaringClass().getSimpleName() + "." + method.getName();
        }
        // key = "SeckillController.seckill"
        
        // 3. 尝试获取令牌
        boolean acquired;
        if (rateLimit.timeoutMs() > 0) {
            acquired = rateLimitService.tryAcquireWithTimeout(key, rateLimit.timeoutMs());
        } else {
            acquired = rateLimitService.tryAcquire(key, rateLimit.qps());
        }
        
        // 4. 没拿到令牌 → 抛异常，请求到此结束
        if (!acquired) {
            throw new IllegalStateException(rateLimit.message());
        }
        
        // 5. 拿到令牌 → 放行，执行真正的业务方法
        return joinPoint.proceed();
    }
}
```

### 执行流程

```
@Around 切面触发
   ↓
读取 @RateLimit 注解的参数（qps、超时时间、提示信息）
   ↓
生成限流器的 key = "类名.方法名"
   ↓
调用 RateLimitService.tryAcquire() 尝试获取令牌
   ↓
┌─────────────────────────────────────┐
│  acquired == true?                   │
│    是 → joinPoint.proceed()          │
│        → 执行 Controller 业务方法     │
│                                     │
│    否 → throw IllegalStateException  │
│        → 请求被拒绝，返回错误提示      │
└─────────────────────────────────────┘
```

## 五、第三层：RateLimitService 限流服务

**文件**：`service/RateLimitService.java`

使用 **Guava RateLimiter** 实现令牌桶算法，底层基于 `SmoothBursty`（平滑突发限流）。

### 令牌桶算法原理

```
┌─────────────────────────────────────────┐
│           令牌桶（Token Bucket）          │
│                                         │
│   令牌以固定速率往桶里放（比如100个/秒）    │
│   桶有容量上限                           │
│   每个请求需要从桶里取一个令牌才能通过      │
│                                         │
│   ┌─────┐                               │
│   │ 🪙🪙🪙 │  ← 桶里当前有令牌            │
│   │ 🪙🪙🪙 │                             │
│   │ 🪙🪙🪙 │                             │
│   └─────┘                               │
│     ↑ ↑ ↑                               │
│   每秒往里放100个                         │
│                                         │
│   请求来了 → 取1个令牌 → 有就通过，没有就拒绝│
└─────────────────────────────────────────┘
```

### 关键设计：每个接口独立限流器

```java
// 存储每个接口的限流器，互不影响
private final Map<String, RateLimiter> rateLimiterMap = new ConcurrentHashMap<>();

private RateLimiter getRateLimiter(String key, double qps) {
    return rateLimiterMap.computeIfAbsent(key, k -> {
        RateLimiter limiter = RateLimiter.create(qps);  // 创建令牌桶
        limiter.tryAcquire();  // 预热：消耗一个令牌，触发生成
        return limiter;
    });
}
```

- `ConcurrentHashMap`：保证多线程安全
- `computeIfAbsent`：key不存在时才创建，保证每个接口只有一个限流器
- `create(qps)`：创建一个每秒产生 `qps` 个令牌的桶
- `tryAcquire()`：预热一下，让桶里立刻有令牌，避免第一个请求被拒绝

### 令牌获取方法

#### 方法1：tryAcquire（带100ms等待）

```java
public boolean tryAcquire(String key, double qps) {
    RateLimiter rateLimiter = getRateLimiter(key, qps);
    // 等待最多100ms，让令牌生成
    return rateLimiter.tryAcquire(100, TimeUnit.MILLISECONDS);
}
```

**为什么等100ms？** 如果直接 `tryAcquire()`（不等待），在限流器刚创建时桶是空的，第一个请求就会被拒绝。等100ms让令牌有机会生成，体验更好。

#### 方法2：tryAcquireWithTimeout（自定义等待时间）

```java
public boolean tryAcquireWithTimeout(String key, long timeoutMs) {
    RateLimiter rateLimiter = getRateLimiter(key, DEFAULT_QPS);
    return rateLimiter.tryAcquire(timeoutMs, TimeUnit.MILLISECONDS);
}
```

#### 方法3：动态调整QPS

```java
public void setRate(String key, double newQps) {
    RateLimiter rateLimiter = getRateLimiter(key, newQps);
    rateLimiter.setRate(newQps);  // 运行时修改令牌生成速率，不需重启
}
```

```
用户请求: GET /seckill/1
        ↓
SeckillController.seckill()
        ↓
RateLimitAspect.around()  ← AOP拦截
        ↓

        ↓
  key = "SeckillController.seckill"
        ↓

        ↓
  RateLimiter.tryAcquire(100ms)
        ↓
  ┌─────────────────────────────┐
  │ 桶里有令牌？                  │
  │   有 → 取走一个，返回true     │
  │   没有 → 等100ms再看          │
  │     100ms内有了 → 返回true    │
  │     超时还没 → 返回false      │
  └─────────────────────────────┘
        ↓
  acquired == true?
        ↓
  true  → joinPoint.proceed() → 执行秒杀业务逻辑
  false → throw IllegalStateException("请求过于频繁，请稍后重试")
              ↓
          Controller 捕获异常 → 返回 {"code": 500, "message": "请求过于频繁，请稍后重试"}
```

## 七、Guava RateLimiter 核心原理

### SmoothBursty（平滑突发限流）

Guava RateLimiter 默认使用 `SmoothBursty` 模型：

```
时间轴（每秒产生100个令牌）：
  |----|----|----|----|----|----|----|----|
  0s   0.1s 0.2s 0.3s 0.4s 0.5s ...  1s
  
  令牌产生速率 = 100个/秒 = 每10ms产生1个令牌
```

### 关键参数

| 参数 | 说明 |
|------|------|
| `permitsPerSecond` | 每秒产生的令牌数（即QPS） |
| `storedPermits` | 当前桶中存储的令牌数 |
| `maxPermits` | 桶的最大容量（SmoothBursty 默认 = permitsPerSecond） |
| `nextFreeTicketMicros` | 下一个令牌可用的时间 |

### tryAcquire 的内部流程

```
tryAcquire(timeout=100ms)
   ↓
计算需要等待的时间 = (1 / permitsPerSecond) × 需要的令牌数
   ↓
如果等待时间 <= timeout → 等待后获取令牌 → 返回true
如果等待时间 > timeout → 直接返回false
```

### 与其他限流算法的对比

| 算法 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **令牌桶** | 固定速率放令牌，取令牌通过 | 允许突发流量，平滑限流 | 实现较复杂 | API限流、秒杀 |
| **漏桶** | 固定速率处理请求，队列缓冲 | 流量绝对平滑 | 不允许突发 | 流量整形 |
| **计数器** | 固定时间窗口内计数 | 实现最简单 | 有临界突发问题 | 简单场景 |
| **滑动窗口** | 细粒度时间窗口计数 | 解决临界问题 | 内存占用高 | 精确限流 |

## 八、设计亮点总结

| 特点 | 实现方式 | 好处 |
|------|----------|------|
| **声明式** | `@RateLimit` 注解 | 加一行注解就有限流，零侵入业务代码 |
| **解耦** | AOP切面 | 限流逻辑和业务逻辑完全分离，各自独立维护 |
| **细粒度** | 每个方法独立的RateLimiter | 接口A限流100不影响接口B限流200 |
| **可配置** | 注解参数 | qps、超时、提示信息都能自定义 |
| **动态调整** | `setRate()` 方法 | 运行时调整某个接口的QPS，不需重启服务 |
| **线程安全** | ConcurrentHashMap + Guava内部同步 | 多线程并发访问安全 |
| **平滑限流** | SmoothBursty模型 | 允许短时突发，整体平稳 |

## 九、QPS配置建议

| 场景 | QPS | 说明 |
|------|-----|------|
| **开发环境** | 10 | 方便测试，容易触发限流 |
| **测试环境** | 100 | 模拟生产压力 |
| **生产环境** | 1000+ | 根据服务器性能和业务量调整 |
| **秒杀场景** | 根据库存 | 库存100，QPS设100-200，避免超卖 |

### 动态调整示例

```java
// 运行时调整某个接口的QPS
rateLimitService.setRate("SeckillController.seckill", 200);  // 调整为200 QPS
```

## 十、测试验证

### 测试方法

```bash
# 1. 用JMeter测试
# 线程数：200
# Ramp-Up：0秒（瞬间并发）
# 循环次数：1

# 2. 预期结果
# - 前100个请求成功（返回200）
# - 后100个请求被限流（返回500，message="请求过于频繁，请稍后重试"）

# 3. 查看RabbitMQ
# 消息数量应该 <= 100（被限流的请求不会发送MQ消息）
```

### 验证要点

1. **QPS限制**：200并发请求，只有约100个能通过
2. **响应提示**：被限流的请求返回 `"请求过于频繁，请稍后重试"`
3. **无MQ消息**：被限流的请求不会进入秒杀流程，MQ队列中无对应消息
4. **恢复能力**：限流触发后，等待1秒再发请求，应该能正常通过
