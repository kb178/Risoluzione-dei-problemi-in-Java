# 本地消息表方案实现说明

## 一、核心思想

**本地消息表**是一种保证数据最终一致性的方案，核心思想是：

> 把"业务操作"和"消息记录"放在**同一个数据库事务**中，保证原子性。

这样即使宕机，重启后也能从消息表中恢复未发送的消息，保证消息最终一定会被发送出去。

## 二、解决的问题

### 2.1 原方案的问题

```
用户请求 → Redis扣减库存 → 发送MQ消息 → 返回
```

**问题场景：Redis扣减成功，但MQ发送前宕机**

| 存储 | 状态 | 说明 |
|------|------|------|
| Redis库存 | ✅ 已扣减 | Lua脚本已执行 |
| Redis用户标记 | ✅ 已设置 | 用户无法再次购买 |
| RabbitMQ消息 | ❌ 未发送 | 宕机导致未执行 |
| 数据库订单 | ❌ 未创建 | 消费者未收到消息 |
| 数据库库存 | ❌ 未扣减 | 消费者未处理 |

**结果：Redis库存 < 数据库库存，数据不一致！**

### 2.2 新方案的改进

```
用户请求 → Redis扣减库存 → 【数据库事务：扣库存 + 创建订单 + 插入本地消息表】 → 返回
                                        ↓
                              定时任务扫描本地消息表 → 发送MQ消息 → 消费者处理 → 标记消息已确认
```

**解决的问题：**
- Redis扣减成功但MQ未发送 → 本地消息表记录了消息，定时任务会重试发送
- MQ发送失败 → 定时任务会自动重试（指数退避策略）
- 消费者处理失败 → 消息会重新入队，定时任务不会重复发送

## 三、执行流程详解

### 3.1 用户下单流程（SeckillServiceImpl.seckill）

```java
@Transactional
public String seckill(Long productId, Long userId) {
    // 1. Redis原子扣减库存（Lua脚本）
    Long result = redisService.seckill(stockKey, userBuyKey, 86400);
    // → 快速判断，毫秒级响应

    // 2. 数据库事务开始
    // 2.1 扣减数据库库存
    productMapper.deductStock(productId, 1);

    // 2.2 创建订单
    orderMapper.insertOrder(productId, userId, orderNo, price, 0);

    // 2.3 插入本地消息表
    messageMapper.insert(new LocalMessage(...));

    // 3. 事务提交（以上3步在同一个事务中，要么全成功，要么全失败）

    return "排队中，订单号：" + orderNo;
}
```

### 3.2 定时任务扫描流程（MessageScanner.scanAndSendMessages）

```java
@Scheduled(fixedRate = 1000)  // 每1秒执行一次
public void scanAndSendMessages() {
    // 1. 查询待发送的消息（status=0 且 next_retry_time <= now）
    List<LocalMessage> messages = messageMapper.selectPendingMessages(LocalDateTime.now());

    // 2. 逐条发送到MQ
    for (LocalMessage message : messages) {
        try {
            rabbitTemplate.convertAndSend(..., message.getMessageContent());

            // 3. 发送成功，更新状态为"已发送"
            messageMapper.updateStatusToSent(message.getId());
        } catch (Exception e) {
            // 4. 发送失败，更新重试信息（指数退避）
            int retryCount = message.getRetryCount() + 1;
            if (retryCount >= message.getMaxRetry()) {
                // 超过最大重试次数，标记为"发送失败"
                messageMapper.updateStatusToFailed(message.getId());
            } else {
                // 计算下次重试时间：2^retryCount 秒后
                LocalDateTime nextRetryTime = LocalDateTime.now().plusSeconds((long) Math.pow(2, retryCount));
                messageMapper.updateRetryInfo(message.getId(), nextRetryTime);
            }
        }
    }
}
```

### 3.3 消费者处理流程（OrderConsumer.handleOrder）

```java
@RabbitListener(queues = RabbitMQConfig.ORDER_QUEUE)
@Transactional
public void handleOrder(OrderMessage message) {
    // 1. 幂等性检查（防止重复消费）
    int count = orderMapper.countByUserAndProduct(...);
    if (count > 0) {
        // 标记本地消息为已确认
        markMessageAsConfirmed(message.getOrderNo());
        return;
    }

    // 2. 扣减MySQL库存
    productMapper.decreaseStock(message.getProductId());

    // 3. 插入订单表
    orderMapper.insert(order);

    // 4. 标记本地消息为"已确认"
    markMessageAsConfirmed(message.getOrderNo());
}
```

## 四、数据状态流转

### 4.1 本地消息状态

```
状态0（待发送） → 状态1（已发送） → 状态2（已确认）
      ↓
状态3（发送失败）→ 等待人工处理
```

### 4.2 消息重试策略（指数退避）

| 重试次数 | 等待时间 | 累计等待时间 |
|----------|----------|--------------|
| 1 | 2秒 | 2秒 |
| 2 | 4秒 | 6秒 |
| 3 | 8秒 | 14秒 |
| 4 | 16秒 | 30秒 |
| 5 | 32秒 | 62秒 |

超过5次重试后，标记为"发送失败"，等待人工处理。

## 五、数据库表结构

### 5.1 本地消息表（local_message）

```sql
CREATE TABLE local_message (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    message_id VARCHAR(50) NOT NULL COMMENT '消息唯一ID（用于幂等）',
    message_type VARCHAR(50) NOT NULL COMMENT '消息类型（如：SECKILL_ORDER）',
    business_id VARCHAR(50) NOT NULL COMMENT '业务ID（如：订单号）',
    message_content TEXT NOT NULL COMMENT '消息内容（JSON格式）',
    status TINYINT NOT NULL DEFAULT 0 COMMENT '状态：0-待发送，1-已发送，2-已确认',
    retry_count INT NOT NULL DEFAULT 0 COMMENT '重试次数',
    max_retry INT NOT NULL DEFAULT 5 COMMENT '最大重试次数',
    next_retry_time DATETIME COMMENT '下次重试时间',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_message_id (message_id),
    INDEX idx_status_next_retry (status, next_retry_time)
);
```

## 七、测试场景

### 7.1 正常场景

1. 用户请求秒杀
2. Redis扣减库存成功
3. 数据库事务执行：扣库存 + 创建订单 + 插入消息表
4. 定时任务扫描到消息，发送到MQ
5. 消费者处理，标记消息已确认

### 7.2 宕机场景（Redis扣减后，MQ发送前宕机）

1. 用户请求秒杀
2. Redis扣减库存成功
3. 数据库事务执行：扣库存 + 创建订单 + 插入消息表
4. **宕机！MQ消息未发送**
5. 重启服务
6. 定时任务扫描到未发送的消息
7. 发送MQ消息
8. 消费者处理，标记消息已确认

### 7.3 MQ发送失败场景

1. 用户请求秒杀
2. Redis扣减库存成功
3. 数据库事务执行：扣库存 + 创建订单 + 插入消息表
4. 定时任务扫描到消息，发送MQ
5. **MQ发送失败！**
6. 更新重试信息，2秒后重试
7. 重试成功
8. 消费者处理，标记消息已确认

## 九、优缺点分析

### 9.1 优点

1. **实现简单**：只需要额外的本地消息表和定时任务
2. **可靠性高**：数据库事务保证原子性，定时任务保证消息最终发送
3. **自动恢复**：宕机后重启，定时任务自动扫描未发送的消息
4. **指数退避**：避免频繁重试导致系统压力
5. **幂等性**：通过messageId保证消息不重复发送

### 9.2 缺点

1. **延迟**：定时任务每隔1秒扫描一次，有秒级延迟
2. **额外存储**：需要本地消息表存储消息记录
3. **消息确认**：需要消费者处理完成后标记消息已确认

## 十、适用场景

1. **订单系统**：下单后发送MQ消息，保证消息最终发送
2. **库存系统**：库存变更后同步到其他系统
3. **积分系统**：订单完成后发送MQ消息，异步增加积分
4. **通知系统**：订单状态变更后发送通知

## 十一、与其他方案对比

| 方案 | 实现复杂度 | 可靠性 | 延迟 | 适用场景 |
|------|------------|--------|------|----------|
| **本地消息表** | 低 | 高 | 秒级 | 普通业务系统 |
| **RocketMQ事务消息** | 中 | 高 | 毫秒级 | 阿里系技术栈 |
| **Saga模式** | 高 | 高 | 秒级 | 跨服务长事务 |
| **TCC** | 高 | 高 | 毫秒级 | 库存、余额场景 |
