## 漏洞 1：补偿服务会导致超卖

### 问题描述

`CompensationService.java:98-103`，当检测到 `redisStock > dbStock` 时，直接以 Redis 库存为准更新数据库：

```java
else if (redisStock > dbStock) {
    int diff = redisStock - dbStock;
    compensateDatabase(productId, redisStock);  // 直接覆盖 DB 库存
}
```

### 原因分析

`redisStock > dbStock` 不一定代表"DB 扣少了"，更常见的情况是 **MQ 里还有消息正在排队等消费**：

1. 用户 A 秒杀成功 → Redis 扣减库存 → 消息发送到 MQ
2. MQ 消息还在队列中等待消费（DB 还没扣减）
3. 此时补偿服务运行，发现 Redis 库存 < DB 库存，判定为"DB 少扣了"
4. 补偿服务把 DB 库存更新为 Redis 的值 → 等于把 MQ 中待处理的订单对应的库存"加回去"了
5. MQ 消费时再扣减 DB → 实际发出的订单数超过了原始库存 → **超卖**

### 解决方案

补偿服务在判断不一致时，应该结合 MQ 队列中待处理的消息数量来综合判断，而不是简单地以 Redis 为准覆盖 DB。

**方案 A：以 DB 为准，反向同步 Redis**

```java
private void analyzeAndCompensate(Long productId, int redisStock, int dbStock) {
    if (redisStock > dbStock) {
        // 不要直接改 DB，应该以 DB 为准回滚 Redis
        // 因为差异可能是 MQ 待处理消息导致的正常现象
        System.out.printf("商品[%d] Redis库存偏高，可能是MQ待处理消息，跳过补偿%n", productId);
        return;
    }

    if (redisStock < dbStock) {
        // 差异在合理范围内（MQ 排队中），不需要处理
        System.out.printf("商品[%d] 库存差异在MQ处理范围内，跳过%n", productId);
        return;
    }
}
```

**方案 B：精确补偿，扣除 MQ 中待处理数量**

```java
private void analyzeAndCompensate(Long productId, int redisStock, int dbStock) {
    // 查询 MQ 队列中该商品待处理的消息数量
    int pendingCount = getPendingMessageCount(productId);

    // 实际应有库存 = DB库存 - MQ待处理数量
    int expectedStock = dbStock - pendingCount;

    if (redisStock != expectedStock) {
        // 只补偿真正的差额
        compensateDatabase(productId, redisStock + pendingCount);
    }
}
```

**推荐方案 A**，实现简单且安全。

## 漏洞 2：用户标记过期后可重复购买

### 问题描述

`SeckillServiceImpl.java:42`，Lua 脚本中用户购买标记 `userBuyKey` 设置了 24 小时过期：

```java
Long result = redisService.seckill(stockKey, userBuyKey, 86400);  // 24小时过期
```

### 原因分析

如果 MQ 消费失败后反复重试，或者消费者处理时间较长，超过 24 小时后 `userBuyKey` 过期删除。此时 Lua 脚本中 `exists userBuyKey` 返回 0，校验不生效，**同一用户可以再次秒杀同一商品，重复下单**。

### 解决方案

**方案 A：延长过期时间**

根据实际业务场景设置合理的过期时间，比如 7 天：

```java
Long result = redisService.seckill(stockKey, userBuyKey, 604800);  // 7天
```

**方案 B：不在 Lua 脚本中设置过期，由 DB 保证唯一性**

在 Lua 脚本中去掉 `userBuyKey` 的过期设置，改为在 `OrderConsumer` 消费时通过数据库唯一索引保证不重复：

```sql
-- 数据库添加唯一索引
ALTER TABLE order_info ADD UNIQUE INDEX uk_user_product (user_id, product_id);
```

```java
// OrderConsumer 中捕获唯一索引冲突异常
try {
    orderMapper.insert(order);
} catch (DuplicateKeyException e) {
    // 重复订单，直接跳过
    System.out.println("重复订单，跳过");
}
```

Lua 脚本中的 `userBuyKey` 改为不过期（或设很长），仅作为前端快速拦截，DB 唯一索引作为最终保障。

**推荐方案 B**，双重保障更可靠。

### ✅ 已修复（2026-07-17）

**修复方案：** 采用方案A，延长过期时间到7天

**修改的文件：**

1. **SeckillServiceImpl.java**：将用户标记过期时间从24小时改为7天

```java
// 修复前：24小时过期
Long result = redisService.seckill(stockKey, userBuyKey, 86400);

// 修复后：7天过期（604800秒）
Long result = redisService.seckill(stockKey, userBuyKey, 604800);
```

**修复效果：** 用户标记从24小时延长到7天，降低过期后重复购买的概率。

## 漏洞 3：MQ 库存扣减失败不回滚 Redis

### 问题描述

`OrderConsumer.java:48-50`：

```java
int updateRows = productMapper.decreaseStock(message.getProductId());
if (updateRows == 0) {
    return;  // 正常返回，自动 ack
}
```

### 原因分析

当 MySQL 库存扣减失败（`updateRows == 0`）时，消息被正常 ack 消费掉了，但 **Redis 的库存和用户购买标记都不会回滚**。结果是：

- 用户被标记为"已购买"（`userBuyKey` 存在）→ 无法再次购买
- 实际没有生成订单 → 用户没买到东西
- Redis 库存比实际可售库存少 → 可售库存被"吃掉"

### 解决方案

在库存扣减失败时，回滚 Redis 库存和用户标记：

```java
@RabbitListener(queues = RabbitMQConfig.ORDER_QUEUE)
@Transactional
public void handleOrder(OrderMessage message) {
    // 1. 幂等性检查
    int count = orderMapper.countByUserAndProduct(message.getUserId(), message.getProductId());
    if (count > 0) {
        return;
    }

    // 2. 扣减 MySQL 库存
    int updateRows = productMapper.decreaseStock(message.getProductId());
    if (updateRows == 0) {
        // 库存不足，回滚 Redis
        String stockKey = "product:stock:" + message.getProductId();
        String userBuyKey = "user:buy:" + message.getProductId() + ":" + message.getUserId();
        redisService.rollback(stockKey, userBuyKey);
        System.out.printf("商品[%d]库存不足，已回滚Redis%n", message.getProductId());
        return;
    }

    // 3. 插入订单表 ...（后续逻辑不变）
}
```

注意：需要在 `OrderConsumer` 中注入 `RedisService`。

## 漏洞 4：rollback 方法非原子操作

### 问题描述

`RedisService.java:69-72`：

```java
public void rollback(String stockKey, String userBuyKey) {
    stringRedisTemplate.opsForValue().increment(stockKey);   // 第1步
    stringRedisTemplate.delete(userBuyKey);                   // 第2步
}
```

### 原因分析

`increment` 和 `delete` 是两次独立的 Redis 调用，不是原子操作。如果执行到第 1 步后、第 2 步前宕机：

- 库存已回滚，但用户标记还在 → 用户无法再次购买（但实际库存已恢复）
- 导致 Redis 和 DB 数据不一致，且用户被"锁死"

### 解决方案

使用 Lua 脚本保证原子性：

```java
private static final String ROLLBACK_SCRIPT =
    "local stockKey = KEYS[1] " +
    "local userBuyKey = KEYS[2] " +
    "redis.call('incr', stockKey) " +
    "redis.call('del', userBuyKey) " +
    "return 1";

public void rollback(String stockKey, String userBuyKey) {
    try {
        RedisSerializer<String> serializer = stringRedisTemplate.getStringSerializer();
        stringRedisTemplate.execute((RedisCallback<Long>) connection -> {
            byte[] script = ROLLBACK_SCRIPT.getBytes();
            byte[] key1 = serializer.serialize(stockKey);
            byte[] key2 = serializer.serialize(userBuyKey);
            return connection.eval(script, ReturnType.INTEGER, 2, key1, key2);
        });
    } catch (Exception e) {
        System.out.println("[ERROR] rollback失败: " + e.getMessage());
    }
}
```

## 漏洞 5：队列监控 Redis key 读写不一致

### 问题描述

`QueueMonitorService.java:88` 和 `QueueMonitorService.java:116`：

```java
// monitorQueueDepth() 写入的是这个 key
stringRedisTemplate.opsForValue().set(QUEUE_DEPTH_KEY, String.valueOf(queueDepth));
// QUEUE_DEPTH_KEY = "queue:depth"

// 但 getQueueDepth() 读取的是另一个 key
String depthStr = stringRedisTemplate.opsForValue().get("queue:depth:current");
```

### 原因分析

写入 `queue:depth`，读取 `queue:depth:current`，两个 key 不一样。`getQueueDepth()` 永远读不到 `monitorQueueDepth()` 写入的数据，导致：

- 队列深度永远返回 0
- 扩容/缩容逻辑永远不会触发
- 告警永远不会触发

### 解决方案

统一使用同一个 key：

```java
private static final String QUEUE_DEPTH_KEY = "queue:depth";

private int getQueueDepth() {
    String depthStr = stringRedisTemplate.opsForValue().get(QUEUE_DEPTH_KEY);
    if (depthStr != null) {
        return Integer.parseInt(depthStr);
    }
    return 0;
}
```

## 漏洞 6：补偿服务读到的 DB 库存可能已过期

### 问题描述

`CompensationService.java:44-57`：

```java
List<Product> products = productMapper.selectAll();  // 第1步：查DB
for (Product product : products) {
    String redisStockStr = stringRedisTemplate.opsForValue().get(stockKey);  // 第2步：查Redis
    int redisStock = Integer.parseInt(redisStockStr);
    int dbStock = product.getStock();  // 用的是第1步查到的值
    if (redisStock != dbStock) { ... }
}
```

### 原因分析

`selectAll()` 和后续的 `getStock()` 之间存在时间差。在高并发秒杀场景下：

1. `selectAll()` 查到 DB 库存 = 10
2. 此时有 3 个 MQ 消费成功，DB 库存变为 7
3. `getStock()` 查到 Redis 库存 = 7
4. 补偿服务发现 `redisStock(7) < dbStock(10)`，判定为 MQ 待处理
5. 但实际上 DB 已经被扣减了，库存是正确的

虽然这种情况下补偿服务不会错误操作（跳过了），但如果时序反过来，可能产生误判。

### 解决方案

将查询改为在同一个事务中执行，或者直接用 SQL 查询最新的 DB 库存：

```java
// 改为实时查询单个商品的最新库存
for (Product product : products) {
    String stockKey = STOCK_KEY_PREFIX + product.getId();
    String redisStockStr = stringRedisTemplate.opsForValue().get(stockKey);
    if (redisStockStr == null) continue;

    // 实时查最新库存，不依赖缓存的 product 对象
    Product latestProduct = productMapper.selectById(product.getId());
    if (latestProduct == null) continue;

    int redisStock = Integer.parseInt(redisStockStr);
    int dbStock = latestProduct.getStock();
    if (redisStock != dbStock) {
        analyzeAndCompensate(product.getId(), redisStock, dbStock);
    }
}
```

## 漏洞 7：rollback 时 key 不存在会导致库存凭空增加

### 问题描述

`RedisService.java:70`：

```java
public void rollback(String stockKey, String userBuyKey) {
    stringRedisTemplate.opsForValue().increment(stockKey);  // 如果 key 不存在会怎样？
    stringRedisTemplate.delete(userBuyKey);
}
```

### 原因分析

Redis 的 `INCR` 命令：如果 key 不存在，会先初始化为 0 再 +1，结果为 1。

如果因为某种原因（比如 Redis 重启、key 被手动删除）`stockKey` 不存在，执行 `increment` 后库存会变成 1，**凭空多出一个库存**。

### 解决方案

在 rollback 前检查 key 是否存在：

```java
private static final String ROLLBACK_SCRIPT =
    "local stockKey = KEYS[1] " +
    "local userBuyKey = KEYS[2] " +
    "if redis.call('exists', stockKey) == 1 then " +
    "    redis.call('incr', stockKey) " +
    "end " +
    "redis.call('del', userBuyKey) " +
    "return 1";

public void rollback(String stockKey, String userBuyKey) {
    try {
        RedisSerializer<String> serializer = stringRedisTemplate.getStringSerializer();
        stringRedisTemplate.execute((RedisCallback<Long>) connection -> {
            byte[] script = ROLLBACK_SCRIPT.getBytes();
            byte[] key1 = serializer.serialize(stockKey);
            byte[] key2 = serializer.serialize(userBuyKey);
            return connection.eval(script, ReturnType.INTEGER, 2, key1, key2);
        });
    } catch (Exception e) {
        System.out.println("[ERROR] rollback失败: " + e.getMessage());
    }
}
```

## 漏洞 8：Integer.parseInt 无异常处理

### 问题描述

`CompensationService.java:56`、`QueueMonitorService.java:72,118`：

```java
int redisStock = Integer.parseInt(redisStockStr);  // 如果不是数字？
```

```java
}
```

## 漏洞 9：currentConsumerCount 多线程不安全

### 问题描述

`QueueMonitorService.java:46`：

```java
private int currentConsumerCount = 2;  // 普通 int
```

### 原因分析

`@Scheduled` 定时任务和手动扩容/缩容可能在不同线程执行，对 `currentConsumerCount` 的读写没有同步保护，可能出现：

- 两个线程同时读到旧值，各自 +2，导致实际扩容了 4 个
- 读到中间状态，判断逻辑出错

### 解决方案

使用 `AtomicInteger`：

```java
private AtomicInteger currentConsumerCount = new AtomicInteger(2);
```

```java
// 读取
int count = currentConsumerCount.get();

// 写入
currentConsumerCount.set(targetCount);

// CAS 操作
currentConsumerCount.compareAndSet(current, target);
```

## 漏洞 10：库存初始化只写一次，重启后不更新

### 问题描述

`StockInitializer.java:35-40` 和 `RedisService.java:77-81`：

```java
// StockInitializer
if (currentStock == null) {
    redisService.initStock(stockKey, product.getStock());
}

// RedisService.initStock
public void initStock(String stockKey, int stock) {
    String currentStock = stringRedisTemplate.opsForValue().get(stockKey);
    if (currentStock == null) {  // 只在 key 不存在时写入
        stringRedisTemplate.opsForValue().set(stockKey, String.valueOf(stock));
    }
}
```

### 原因分析

`initStock` 只在 Redis 中没有这个 key 时才写入。场景：

1. 第一次启动，Redis 无数据 → 初始化 Redis 库存 = 100
2. 秒杀过程中，Redis 库存扣减到 50，DB 库存扣减到 50
3. 重启应用，`StockInitializer` 运行，但 Redis 中已有 `product:stock:1 = 50`
4. `currentStock != null` → 跳过初始化
5. 但如果此时 DB 库存已经被补偿服务改回了 100，**Redis 和 DB 就不一致了**

更严重的情况：如果 Redis 被清空重启，但 DB 库存还是上次的值，`StockInitializer` 会用 DB 的当前值初始化，但如果之前有 MQ 消息还在排队，初始化值就会偏大。

### 解决方案

每次启动时以 DB 为准强制覆盖 Redis：

```java
// StockInitializer.java
public void run(String... args) throws Exception {
    List<Product> products = productMapper.selectAll();
    for (Product product : products) {
        String stockKey = STOCK_KEY_PREFIX + product.getId();
        // 每次启动都用 DB 的值覆盖 Redis
        stringRedisTemplate.opsForValue().set(stockKey, String.valueOf(product.getStock()));
        System.out.printf("商品[%s]库存已同步到Redis: %d%n", product.getName(), product.getStock());
    }
}
```

或者如果需要保留 Redis 中已扣减的库存（避免重启后库存"回弹"），应该用 Lua 脚本只在 key 不存在时写入：

```java
private static final String INIT_STOCK_SCRIPT =
    "if redis.call('exists', KEYS[1]) == 0 then " +
    "    redis.call('set', KEYS[1], ARGV[1]) " +
    "    return 1 " +
    "end " +
    "return 0";
```

## 漏洞 11：decreaseStock 成功但 insert 失败，消息被 ack

### 问题描述

`OrderConsumer.java:46-62`：

```java
// 第2步：扣减库存（成功）
int updateRows = productMapper.decreaseStock(message.getProductId());
if (updateRows == 0) { return; }

// 第3步：插入订单（如果这里抛异常）
orderMapper.insert(order);
```

### 原因分析

即使 `@Transactional` 正常工作，存在以下时序问题：

1. `decreaseStock` 执行成功（DB 库存 -1）
2. `insert` 抛异常（比如字段超长、类型不匹配等）
3. `@Transactional` 回滚，DB 库存恢复
4. 异常向上抛出 → AUTO 模式 nack → 消息重新入队
5. 消息重新消费时，`decreaseStock` 再次执行 → **库存被多扣一次**

根本原因：**数据库事务回滚 和 RabbitMQ 消息重试 不是原子的**。事务回滚了 DB，但消息已经离开了队列，重新入队后再消费就会重复扣减。

### 解决方案

在消息处理中加入版本号或乐观锁，防止重复扣减：

**方案 A：用订单号做幂等键**

```java
@RabbitListener(queues = RabbitMQConfig.ORDER_QUEUE)
public void handleOrder(OrderMessage message) {
    // 先检查订单是否已存在（幂等检查）
```

## 漏洞 12：@Transactional 可能对 @RabbitListener 不生效

### 问题描述

`OrderConsumer.java:31`：

```java
@RabbitListener(queues = RabbitMQConfig.ORDER_QUEUE)
@Transactional
public void handleOrder(OrderMessage message) { ... }
```

### 原因分析

Spring 的 `@Transactional` 基于 AOP 代理实现。但 `@RabbitListener` 的调用链是：

```
RabbitMQ消息 → ListenerContainer → 反射调用handleOrder()
```

在某些 Spring 版本或配置下，ListenerContainer 可能直接通过反射调用目标方法，**绕过了 AOP 代理**，导致 `@Transactional` 不生效。表现为：

- `decreaseStock` 和 `insert` 不在同一个事务中
- `decreaseStock` 成功后立即提交
- `insert` 失败时，库存已经扣了但订单没插入
- **库存凭空消失，用户没买到东西**

### 解决方案

**方案 A：手动管理事务**

```java
@Autowired
private PlatformTransactionManager transactionManager;

@RabbitListener(queues = RabbitMQConfig.ORDER_QUEUE)
public void handleOrder(OrderMessage message) {
    TransactionStatus status = transactionManager.getTransaction(
        new DefaultTransactionDefinition());
    try {
        // ... 业务逻辑
        orderMapper.insert(order);
        transactionManager.commit(status);
    } catch (Exception e) {
        transactionManager.rollback(status);
        throw e;  // 抛出让 AUTO 模式 nack
    }
}
```

**方案 B：确保代理生效（推荐）**

在配置类中确保 RabbitListener 容器使用代理：

```java
@Bean
public SimpleRabbitListenerContainerFactory rabbitListenerContainerFactory(
        ConnectionFactory connectionFactory) {
    SimpleRabbitListenerContainerFactory factory = new SimpleRabbitListenerContainerFactory();
    factory.setConnectionFactory(connectionFactory);
    factory.setMessageConverter(jsonMessageConverter());
    // 确保通过代理调用
    factory.setAdviceChain(transactionInterceptor());
    return factory;
}

@Bean
public TransactionInterceptor transactionInterceptor() {
    return new TransactionInterceptor(transactionManager, Collections.singletonMap(
        "*", TransactionDefinition.PROPAGATION_REQUIRED));
}
```

## 漏洞 15：补偿定时任务中 parseInt 异常会中断所有后续商品检查

### 问题描述

`CompensationService.java:44-67`：

```java
@Scheduled(fixedRate = 30000)
public void checkStockConsistency() {
    List<Product> products = productMapper.selectAll();
    
    for (Product product : products) {
        String redisStockStr = stringRedisTemplate.opsForValue().get(stockKey);
        if (redisStockStr == null) continue;
        
        int redisStock = Integer.parseInt(redisStockStr);  // ← 如果这里抛异常
        // ... 后面的商品全部跳过不检查
    }
}
```

### 原因分析

`Integer.parseInt` 没有 try-catch。如果某个商品的 Redis 库存值不是合法数字（比如被写入了 `"null"`、`""`、或被其他服务误写入了字符串），会抛出 `NumberFormatException`。

由于异常在 for 循环内部，**该商品之后的所有商品都不会被检查**。比如有 100 个商品，第 3 个出问题，后面 97 个全部跳过。

而且 `@Scheduled` 方法如果抛出未捕获异常，Spring 会打日志但任务会继续执行，不过本次执行中途就断了。

### 解决方案

在循环内部加 try-catch：

```java
@Scheduled(fixedRate = 30000)
public void checkStockConsistency() {
    List<Product> products = productMapper.selectAll();
    int inconsistentCount = 0;
    
    for (Product product : products) {
        try {
            String stockKey = STOCK_KEY_PREFIX + product.getId();
            String redisStockStr = stringRedisTemplate.opsForValue().get(stockKey);
            if (redisStockStr == null) continue;
            
            int redisStock = Integer.parseInt(redisStockStr);
            int dbStock = product.getStock();
            
            if (redisStock != dbStock) {
                inconsistentCount++;
                analyzeAndCompensate(product.getId(), redisStock, dbStock);
            }
        } catch (NumberFormatException e) {
            System.out.printf("商品[%d] Redis库存值异常: %s，跳过%n", 
                product.getId(), stringRedisTemplate.opsForValue().get(STOCK_KEY_PREFIX + product.getId()));
        } catch (Exception e) {
            System.out.printf("商品[%d] 检查异常: %s，跳过%n", product.getId(), e.getMessage());
        }
    }
    
    System.out.printf("库存一致性检查完成，发现%d个不一致数据%n", inconsistentCount);
}
```

## 漏洞 16：AUTO 模式下消息失败会无限重试，永远不会进死信队列

### 问题描述

`application.properties:22` 和 `OrderConsumer.java:31`：

```properties
spring.rabbitmq.listener.simple.acknowledge-mode=auto
```

```java
@RabbitListener(queues = RabbitMQConfig.ORDER_QUEUE)
@Transactional
public void handleOrder(OrderMessage message) {
    // 如果这里抛异常...
}
```

### 原因分析

AUTO 模式的重试机制：

1. `handleOrder` 抛出异常
2. Spring 自动 nack 消息，设置 `requeue=true`
3. 消息重新入队，再次被消费
4. 如果还是抛异常 → 回到第 1 步
5. **无限循环**

虽然 `RabbitMQConfig` 配置了死信队列，但死信队列只在以下情况触发：
- `basicReject/basicNack` 且 `requeue=false`
- 消息 TTL 过期
- 队列达到最大长度

AUTO 模式抛异常时是 `requeue=true`，**消息永远不会进入死信队列**。如果一条消息因为数据问题（比如商品 ID 不存在）持续失败，会无限重试，占用消费者线程，拖垮整个系统。

### 解决方案

**方案 A：配置 Spring 重试机制，达到最大次数后拒绝**

```properties
# 开启重试
spring.rabbitmq.listener.simple.retry.enabled=true
# 最大重试次数（不含首次）
spring.rabbitmq.listener.simple.retry.max-attempts=3
# 重试间隔
spring.rabbitmq.listener.simple.retry.initial-interval=1000
# 最大间隔
spring.rabbitmq.listener.simple.retry.max-interval=10000
# 重试所有异常
spring.rabbitmq.listener.simple.retry.multiplier=1.0
```

重试耗尽后，需要自定义 `RejectAndDontRequeueRecoverer` 把消息发到死信队列：

```java
@Bean
public SimpleRabbitListenerContainerFactory rabbitListenerContainerFactory(
        ConnectionFactory connectionFactory) {
    SimpleRabbitListenerContainerFactory factory = new SimpleRabbitListenerContainerFactory();
    factory.setConnectionFactory(connectionFactory);
    factory.setMessageConverter(jsonMessageConverter());
    factory.setAdviceChain(transactionInterceptor(), retryInterceptor());
    return factory;
}

@Bean
public RetryInterceptorBuilder retryInterceptor() {
    return RetryInterceptorBuilder.stateless()
        .maxAttempts(3)
        .backOffOptions(1000, 2.0, 10000)
        .recoverer(new RejectAndDontRequeueRecoverer());  // 重试耗尽后发到死信队列
}
```

**方案 B：手动模式（MANUAL），代码控制 ACK**

```java
@RabbitListener(queues = RabbitMQConfig.ORDER_QUEUE)
public void handleOrder(OrderMessage message, Channel channel,
                        @Header(AmqpHeaders.DELIVERY_TAG) long deliveryTag) {
    try {
        // 业务逻辑...
        channel.basicAck(deliveryTag, false);  // 成功：ack
    } catch (Exception e) {
        // 重试 3 次后拒绝，不重新入队（进入死信队列）
        channel.basicNack(deliveryTag, false, false);  // false = 不重新入队 → 进死信队列
    }
}
```

```java
@RateLimit(qps = 100, message = "请求过于频繁，请稍后重试")
public Map<String, Object> seckill(@PathVariable Long productId,
                                   @RequestBody Map<String, Long> body) {
    Long userId = body.get("userId");
    
    Map<String, Object> result = new HashMap<>();
    if (productId == null || productId <= 0) {
        result.put("code", 400);
        result.put("message", "商品ID无效");
        return result;
    }
    if (userId == null || userId <= 0) {
        result.put("code", 400);
        result.put("message", "用户ID无效");
        return result;
    }
    
    try {
        String message = seckillService.seckill(productId, userId);
        result.put("code", 200);
        result.put("message", message);
    } catch (Exception e) {
        result.put("code", 500);
        result.put("message", e.getMessage());
    }
    return result;
}
```

```java
public class SeckillRequest {
    @NotNull @Positive
    private Long productId;
    
    @NotNull @Positive
    private Long userId;
}
```

## 总结

| 漏洞 | 根本原因 | 核心修复思路 |
|------|----------|--------------|
| 超卖 | 补偿服务误判差异原因 | 以 DB 为准，不盲目覆盖 |
| 重复购买 | 用户标记有 TTL 且无 DB 兜底 | DB 唯一索引做最终保障 |
| Redis 库存"吃掉" | MQ 消费失败未回滚 Redis | 消费失败时主动回滚 |
| rollback 非原子 | 两次 Redis 调用无事务保证 | 改用 Lua 脚本 |
| 队列监控失效 | Redis key 读写不一致 | 统一使用同一个 key |
| 补偿误判 | DB 库存快照过期 | 实时查询最新库存 |
| 库存凭空增加 | rollback 未检查 key 存在性 | Lua 脚本中先判断 exists |
| 定时任务中断 | parseInt 无异常处理 | 添加 try-catch |
| 线程安全 | 普通 int 多线程读写 | 改用 AtomicInteger |
| 库存初始化不更新 | initStock 只写一次 | 启动时强制同步 DB |
| 事务回滚后消息被 ack | DB 事务与 MQ ACK 非原子 | 订单号做幂等键 |
| @Transactional 失效 | AOP 代理被绕过 | 手动管理事务或配置代理 |
| userId 伪造 | GET 参数可篡改 | 改 POST + 登录态获取 |
| 启动崩溃 | PostConstruct 无异常处理 | 添加 try-catch |
| 检查中断 | 循环内 parseInt 无保护 | 单个商品异常不影响其他 |
| 消息无限重试 | AUTO 模式 requeue=true | 配置重试次数 + RejectAndDontRequeueRecoverer |
| 扩容空实现 | startNewConsumer 只打印日志 | 删除伪逻辑或用 API 实现 |
| 并发竞态 | 幂等检查和插入非原子 | DB 唯一索引 + catch DuplicateKeyException |
| 无参数校验 | productId/userId 无边界检查 | 加 @Positive 或手动校验 |
