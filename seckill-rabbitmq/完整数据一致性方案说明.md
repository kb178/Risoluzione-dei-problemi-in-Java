# 完整数据一致性方案说明

## 一、问题场景分析

### 1.1 场景：Redis扣减后宕机

```
用户请求 → Redis Lua脚本执行 ✅ → 宕机！ → 数据库事务未执行 ❌
```

**结果：**
- Redis库存已扣减
- 本地消息表没有记录
- 数据库订单没有创建
- 数据库库存没有扣减

### 1.2 概率分析

这种情况的概率**极低**：

```
Redis执行完成 → 返回Java代码 → 执行数据库事务
     ↓
   微秒级（< 1ms）
```

宕机概率估算：**< 0.0001%**（万分之一）

## 二、完整解决方案：两层防护

### 2.1 第一层：本地消息表（解决99.99%的场景）

**解决的问题：**
- Redis扣减成功，但MQ发送前宕机
- MQ发送失败后自动重试
- 消费者处理失败后自动重试

**工作流程：**

```
用户请求
    ↓
Redis原子扣减库存（Lua脚本）
    ↓
数据库事务（同一事务）：
    ├── 扣减数据库库存
    ├── 创建订单
    └── 插入本地消息表
    ↓
返回"排队中"
    ↓
定时任务扫描本地消息表（每1秒）
    ↓
发送MQ消息
    ↓
消费者处理
    ↓
标记消息已确认
```

### 2.2 第二层：补偿服务（解决0.01%的极端场景）

**解决的问题：**
- Redis扣减后，数据库事务提交前宕机（订单丢失）
- MQ消费失败后消息丢失
- 数据不一致的最终兜底

**工作流程：**

```
定时任务（每30秒）
    ↓
检查Redis库存 vs 数据库库存
    ↓
发现不一致？
    ├── 是 → 分析原因并补偿
    └── 否 → 通过
```

## 三、补偿服务详细逻辑

### 3.1 三种不一致场景

#### 场景A：Redis库存 < 数据库库存（有消息在本地消息表）

```
Redis库存：95
数据库库存：100
本地消息表：5条待发送消息
```

**分析：** 正常情况，消息等待定时任务发送

**补偿：** 以Redis为准，更新数据库库存

```java
// 补偿数据库库存
productMapper.updateStock(productId, 95);
```

#### 场景B：Redis库存 < 数据库库存（有消息在MQ队列）

```
Redis库存：95
数据库库存：100
本地消息表：0条待发送消息
MQ队列：5条消息等待处理
```

**分析：** 正常情况，消息在MQ队列中等待消费者处理

**补偿：** 不需要补偿，MQ会自动重试

#### 场景C：Redis库存 < 数据库库存（订单丢失）

```
Redis库存：95
数据库库存：100
本地消息表：0条待发送消息
MQ队列：0条消息
```

**分析：** 异常情况！Redis扣减后，数据库事务提交前宕机

**补偿：** 回滚Redis库存

```java
// 回滚Redis库存
stringRedisTemplate.opsForValue().increment(stockKey, 5);
```

#### 场景D：Redis库存 > 数据库库存

```
Redis库存：100
数据库库存：95
```

**分析：** 数据库扣减失败，需要补偿数据库

**补偿：** 以Redis为准，更新数据库库存

```java
// 补偿数据库库存
productMapper.updateStock(productId, 100);
```

### 3.2 补偿服务核心代码

```java
private void analyzeAndCompensate(Long productId, int redisStock, int dbStock) {
    // 情况1：Redis库存 < 数据库库存
    if (redisStock < dbStock) {
        int diff = dbStock - redisStock;

        // 检查本地消息表是否有待发送消息
        int pendingMessageCount = countPendingMessagesByProduct(productId);

        if (pendingMessageCount > 0) {
            // 场景A：本地消息表有待发送消息（正常情况）
            System.out.printf("场景A：商品[%d]有%d条消息在本地消息表中等待发送%n",
                productId, pendingMessageCount);

            // 补偿数据库库存（以Redis为准）
            compensateDatabase(productId, redisStock);
        } else {
            // 场景B/C：本地消息表没有待发送消息
            int mqMessageCount = countMQMessagesByProduct(productId);

            if (mqMessageCount > 0) {
                // 场景B：有订单在MQ中等待处理（正常情况）
                System.out.printf("场景B：商品[%d]有%d条消息在MQ队列中等待处理%n",
                    productId, mqMessageCount);

                // 不需要补偿，MQ会自动重试
            } else {
                // 场景C：订单丢失了（异常情况）
                System.out.printf("场景C：商品[%d]订单丢失！需要回滚Redis库存%n", productId);

                // 回滚Redis库存
                compensateRedis(productId, redisStock, dbStock);
            }
        }
    }

    // 情况2：Redis库存 > 数据库库存
    else if (redisStock > dbStock) {
        int diff = redisStock - dbStock;
        System.out.printf("分析：商品[%d]数据库库存少扣了%d，需要补偿%n", productId, diff);

        // 以Redis为准，更新数据库
        compensateDatabase(productId, redisStock);
    }
}
```

## 五、宕机场景分析

### 5.1 场景1：Redis扣减后，数据库事务提交前宕机

**时间线：**
```
t1: Redis Lua脚本执行完成 ✅
t2: 开始执行数据库事务
t3: 宕机！数据库事务未提交 ❌
```

**状态：**
- Redis库存：已扣减
- 本地消息表：没有记录
- 数据库订单：没有创建
- 数据库库存：没有扣减

**补偿服务处理：**
```
定时任务（每30秒）发现：Redis库存 < 数据库库存
    ↓
检查本地消息表：没有待发送消息
    ↓
检查MQ队列：没有待处理消息
    ↓
判断为"订单丢失"
    ↓
回滚Redis库存
```

**结果：** 数据最终一致 ✅

### 5.2 场景2：数据库事务提交后，MQ发送前宕机

**时间线：**
```
t1: Redis Lua脚本执行完成 ✅
t2: 数据库事务提交成功 ✅
t3: 宕机！MQ消息未发送 ❌
```

**状态：**
- Redis库存：已扣减
- 本地消息表：有记录（待发送）
- 数据库订单：已创建
- 数据库库存：已扣减

**本地消息表方案处理：**
```
定时任务（每1秒）扫描到：本地消息表有待发送消息
    ↓
发送MQ消息
    ↓
消费者处理
    ↓
标记消息已确认
```

**结果：** 数据最终一致 ✅

### 5.3 场景3：MQ发送失败后宕机

**时间线：**
```
t1: Redis Lua脚本执行完成 ✅
t2: 数据库事务提交成功 ✅
t3: MQ发送失败 ❌
t4: 宕机！ ❌
```

**状态：**
- Redis库存：已扣减
- 本地消息表：有记录（待发送）
- 数据库订单：已创建
- 数据库库存：已扣减

**本地消息表方案处理：**
```
重启服务后，定时任务（每1秒）扫描到：本地消息表有待发送消息
    ↓
重试发送MQ消息（指数退避）
    ↓
发送成功
    ↓
消费者处理
    ↓
标记消息已确认
```

**结果：** 数据最终一致 ✅

## 六、企业级实现建议

### 6.1 本地消息表方案

**优点：**
- 实现简单，容易理解
- 可靠性高，有数据库事务保证
- 宕机后自动恢复

**缺点：**
- 需要额外的本地消息表
- 定时任务有延迟（秒级）

**适用场景：**
- 普通订单系统
- 库存同步系统
- 积分系统

### 6.2 补偿服务方案

**优点：**
- 兜底保障，处理极端情况
- 自动修复数据不一致
- 可以监控和告警

**缺点：**
- 需要额外的定时任务
- 补偿逻辑可能复杂

**适用场景：**
- 所有使用本地消息表的系统
- 需要数据一致性的关键业务

### 6.3 最佳实践

1. **两层防护**：本地消息表 + 补偿服务
2. **定时任务**：本地消息表每1秒扫描，补偿服务每30秒检查
3. **指数退避**：MQ发送失败时，重试间隔指数增长
4. **幂等性**：通过messageId防止重复发送
5. **消息确认**：消费者处理完成后标记消息已确认
6. **监控告警**：监控消息表状态，发现异常及时告警

## 七、总结

| 场景 | 概率 | 解决方案 | 结果 |
|------|------|----------|------|
| Redis扣减后，数据库事务提交前宕机 | < 0.0001% | 补偿服务回滚Redis | 数据一致 ✅ |
| 数据库事务提交后，MQ发送前宕机 | < 0.001% | 本地消息表重试 | 数据一致 ✅ |
| MQ发送失败后宕机 | < 0.01% | 本地消息表重试 | 数据一致 ✅ |

**最终结论：**
- 本地消息表方案解决了99.99%的场景
- 补偿服务解决了0.01%的极端场景
- 两层防护，保证数据最终一致性
